# 七阶段定义

骨架不许调换顺序。每一阶段的 HTML 结构和样式约定见 `renderers/html-shell.md`，完整模板见 `templates/`。

**设计依据见 `core/learning-science.md`**：CTML 预训练 → §1；具体化 + 生成活动 + 脚手架 → §3；脚手架 + 模态 → §4；个性化 + 生成活动 → §5–§7。

---

## Phase 1 — 知识点清单 + 知识地图

**目标**：让用户在读具体内容之前就知道"这章有多少东西、它们之间什么关系"。

**做**：
- 穷尽本章所有知识点（定义/公式/定理/例题/图/脚注）
- PPT 出现过的用 🎯 标记
- **每个知识点附一句"一句话定义"** —— 不展开，只先命名。格式：`1. [知识点名]（原书 §X.Y）🎯 — 一句话定义。`
  - 依据：CTML 预训练原则——先命名概念再深入讲解，学习者后续加工效率更高
  - 一句话定义标准：用日常语言，不包含多于 1 个术语；如果必须用术语，紧接着用括号解释
- 清单之后画一张 SVG 知识地图，展示所有知识点的层级和依赖关系
- 地图中每个节点需标注：上游（依赖什么）、下游（打开什么）、邻居（并列什么）

**不做**：
- 不跳过任何原书脚注或边栏注释
- 不省略"简单"的概念——VSL 用户需要全局视野
- 一句话定义不展开——展开是 §3–§5 的事

**质量标准**：
- 清单条目数 ≥ 原书小节数（通常更多，因为一个小节可能有多个概念）
- 知识地图覆盖清单所有条目
- 每个知识点都有一句话定义
- 一句话定义不包含多于 1 个术语

## Phase 2 — 逆向目标

**目标**：让用户知道"学完能干什么"，而不是"要学什么"。

**做**：
- 从例题/习题/PPT 练习中提取 3–6 条可检验的能力陈述
- 每条格式："给定 [输入]，能 [动作]，并且 [质量约束]"
- 必须对应到原书具体出处

**不做**：
- 禁止"理解/掌握/了解/熟悉"这类不可检验动词
- 不超过 6 条——太多记不住

**质量标准**：
- 覆盖本章主要例题类型
- 每条能力陈述都可以用一道题验证

---

## Phase 3 — 心智模型构建（核心是什么 + 怎么操作 + 怎么生长）

**目标**：在学生脑子里种一棵可以挂东西的树。不是讲学科史，不是列概念清单。学生读完 §3 时，应该能用自己的话说出：① 这章最核心的一个概念是什么 ② 这个概念的操作直觉是什么 ③ 本章其他关键概念怎么从这个核心长出来。

**做**：
- §3.1 **核心概念锚定**（1-2 句话）：直接告诉学生这章最核心的一个概念是什么。用世界知识解释，不用原书语言。不超过 25 个字，不含多于 1 个术语。
  - 依据：CTML 预训练 + 具体化原则——先给核心锚点，再展开
- §3.2 **操作直觉**（一句话可执行）：给一个操作层面的直觉，学生看到这个概念就能用、能算、能判断。动词开头，读完能直接操作。
  - 依据：Mayer 生成活动原则——操作直觉比定义更容易激活生成加工
- §3.3 **心智模型生长**（从核心展开到全章）：从 §3.1 的核心概念出发，**用第一性原理推导**展开到本章所有关键概念。每个节点：① 回到核心建立连接 ② 问"现有工具够用吗？不够就必须发明新概念" ③ 给操作直觉 ④ 用世界知识解释。用推导链（flow-container 或 SVG）展示，节点数 ≤ 8。
  - 每个箭头是第一性原理推导（"现有工具不够用，必须发明 B"），不是简单连接（"然后呢"）
  - 推导方法详见 `methods/first-principles.md`，生长结构详见 `methods/mental-model.md`
  - 最后一个节点覆盖本章核心内容（§4 精读的主要部分）
  - 关键转折步骤紫色高亮标注"🔑 关键转折"——这是新概念诞生的时刻
  - 依据：Mayer 信号原则——步骤编号降低视觉搜索成本，迁移测试得分提升 d=0.58
- §3.4 在知识框架中的位置（SVG 树形图 / 对照表）
- §3.5 **暂停思考**（元认知提示）：

```
> 🧠 **暂停思考**
>
> 现在合上书（或者滚到屏幕外面去），问自己：
> - 你能用自己的话说"这章最核心的概念是什么"吗？
> - [核心概念]的操作是什么？（一句话）
> - §3.3 的生长链里，哪个节点你觉得"为什么要这样"？
>
> 如果说不出来，把对应节点重新读一遍再继续。
> —— §4 会用原书的细节来填充这个框架。但先确保骨架是稳的。
```

**不做**：
- 不讲学科史（"19 世纪数学家发现……"）——那是背景，不是心智模型
- 不列概念清单——那是 §1 的事
- 不宣布结论——要让学生看到概念从核心"长出来"
- 不展开公式怎么用、怎么做题——这些是 §4 的事
- 不用原书语言解释概念——用世界知识，原书语言留给 §4

**质量标准**：
- §3.1 读完后，学生能用一句话回答"这章讲什么"
- §3.2 读完后，学生看到概念能直接操作（不需要回去翻定义）
- §3.3 读完后，学生脑子里有一棵完整的"树"——知道本章有哪些关键概念、它们之间什么关系
- §3.3 每个箭头都是第一性原理推导（"现有工具不够用，必须发明 B"），不是简单连接
- 读者读完 §3 的感受是"我知道这章要干什么了"，而不是"我知道这章有什么了"（那是 §1）或"我知道为什么需要这个了"（那是旧版 §3）

## Phase 4 — 场景式精读（用原书内容填充 §3 的心智模型）

**目标**：§3 已经搭好了心智模型（树），§4 是用原书的内容往树上挂叶子。§4 完成后，学生**不需要再打开原书**——所有内容都在课件里，而且比原书更容易理解。

### §4 的本质：还原作者脑中的图景

作者写书时，脑子里有一个关于这部分知识的**图景**（比如生日问题的场景、文氏图的重叠、条件概率的缩小空间）。作者用文字把这个图景描述出来。

§4 要做的事情：
1. 把原书文字按**图景**分块（不是按段落分）
2. 每块先展示原书的描述（整块，不是金句摘抄）
3. 然后告诉学生"这段文字在描述什么图景"
4. 如果有图，直接嵌入原图
5. 用 SVG 画出这个图景的示意图
6. 如果有公式推导，从零开始，每一步都解释

### 做

**A. 分块原则——按图景切分，不是按段落**

- 原书中描述**同一个图景/概念/例子**的文字归为一块
- 每块的标题是"这个图景是什么"（不是原书的小节号）
- 示例：原书 §4.1 可能包含"条件概率的定义"和"钓鱼的例子"两个图景，分成两块
- 场景数量要足够覆盖原书所有内容——宁可多分几个场景，不要漏掉任何知识点

**B. 原文呈现——整块展示，不是金句摘抄**

每块包含：
- **原书文字**：取这块的**首句、尾句、中间关键内容**。不必逐字抄写（HTML 会太长），但必须**覆盖原书所有知识点**，确保学生不需要再打开原书
  - 引用格式：用 `<blockquote>` 包裹，保持原书语言不翻译
  - 长段落：取首句 + 尾句 + 中间关键句子，用 `……` 省略过渡内容
  - 关键原则：**所有摘抄的段落组合起来能覆盖原书全部内容**
- **图景解说**：这段文字在描述什么？作者脑子里的画面是什么？用你自己的话，让学生"身临其境"
  - **不要吝啬解说——要慷慨地详细展开**。解说不是 1-2 句话的概括，而是要像站在学生旁边，指着原文说"你看，作者这里在讲的是……"
  - 解说要回答：这个概念/机制/过程**具体是怎么工作的**？每一步发生了什么？
  - 对于技术类内容，解说要包含**完整的调用链/执行流/数据流**——从入口到出口，每一步都不跳过
- **原图嵌入**：原书有图就用 `<img>` 嵌入，路径从 MinerU 的 markdown 改写过来（`![[Mathematics/raw/xxx/assert/yyy.jpg]]` → `<img src="../raw/xxx/assert/yyy.jpg">`）
  - **必须嵌入原图**——MinerU 已经导出了图片，agent 不要自己重新画，直接引用
- **SVG 示意图**：如果原书没图但内容需要可视化（比如文氏图、推导流程），画 SVG
  - 原书有图的不需要再画 SVG——用原图即可

**C. 公式推导——从零开始，每一步都解释**

学生已经放空了自己，可能连第一条公式都不会写。推导要求：
- 每个推导步骤标注序号（Step 1/n）
- 每一步解释"为什么要这样做"，不只是"做了什么"
- 如果是从已有公式出发，先写出那个公式，说明它从哪来
- 推导过程中的每一步变换都要有文字说明（不能只列算式）
- 示例（贝叶斯定理推导）：
  ```
  Step 1/4：我们有两个关于 Pr(A∩B) 的表达式（来自乘法法则）
            Pr(A∩B) = Pr(A|B) × Pr(B)    ……①
            Pr(A∩B) = Pr(B|A) × Pr(A)    ……②
            ①来自条件概率的定义，②是把 A 和 B 的角色互换
  Step 2/4：①和②左边相同，所以右边也相等
            Pr(A|B) × Pr(B) = Pr(B|A) × Pr(A)
  Step 3/4：我们想求 Pr(A|B)，所以把 Pr(B) 除过去
            Pr(A|B) = Pr(B|A) × Pr(A) / Pr(B)
  Step 4/4：这就是贝叶斯定理。它的意义：知道"原因→结果"的概率(B|A)，
            反过来求"结果→原因"的概率(A|B)。
  ```

**D. 流程图/推导链——必须有文字解释**

SVG 推导链的每个节点必须有：
- 节点名（概念名）
- 节点下方的文字说明（这个节点在做什么，不只是一个标题）
- 箭头上的文字说明（为什么从这一步到下一步）
- 不能只有框和箭头没有解释

**E. 例题——完整呈现，从读题到答案**

- 完整呈现原书例题的题目（不是只给一句话概括）
- 解题过程从"读题→识别类型→选择方法→逐步计算→验证"全链路
- 每一步标注是原书的方法还是 agent 补充的
- 书外补充用蓝色 callout，PPT 补充用橙色 callout

**F. 覆盖度——覆盖原书所有内容**

- §4 必须覆盖原书对应章节的**所有**知识点、定义、公式、定理、例题、脚注
- 不是选择性摘抄——是完整的替代品
- 学生读完 §4 后，不需要再打开原书
- **自检方法**：把 §4 中所有 `<blockquote>` 的内容拼起来，检查是否覆盖了原书的每一节、每一个知识点

**G. 技术类内容的特殊要求**

当原书是技术类内容（操作系统、编程、计算机系统等）时，§4 还需要：
- **完整呈现源码**：不是只引一两行，而是给出完整的函数实现（带注释）
- **完整调用链**：从用户层到内核层，每一步都追踪，不跳过中间层次
- **常见错误 callout**：用红色 warning callout 列出学生最容易犯的错误（每个场景 2-3 个）
- **关键代码位置**：明确指出"你需要改哪个文件的哪个函数"
- **调试提示**：如果某个实现有陷阱（比如 sbrk 的死递归），必须在 callout 中警告

### 有界生成流程

§4 内容密度最高、HTML 结构最复杂，采用分离模式：
- Agent 先输出结构化 JSON → 检查 schema → `scripts/render-courseware.mjs` 渲染 HTML 片段 → 嵌入主模板
- JSON schema 定义见 `scripts/render-courseware.mjs` 文件头部注释

**§4 完成后强制运行** `scripts/validate-html.mjs`，所有错误修正后进入 §5

### 结尾暂停思考

```
> 🧠 **暂停思考**
>
> 你刚读完本章最硬核的部分。在进入费曼讲法之前：
> - 不看屏幕，回忆 §3 的心智模型：核心是什么？操作直觉是什么？生长链怎么走？
> - 然后试着把 §4 的细节挂到对应的节点上——哪个节点挂得住？哪个还空着？
> - 如果某个节点挂不住 → 回去看对应的 §4 场景。
> - 如果都挂住了 → 好，§5 费曼讲法来压测一下。
```

### 不做

- **不搞金句摘抄**——原书的每一段都要覆盖到，不是只引几句"经典"。所有引用拼起来要能覆盖原书全部内容
- **不吝啬解说**——图景解说不是 1-2 句话的概括，要像站在学生旁边详细展开。不要不舍得多讲几句
- **不跳过原图**——MinerU 已经导出了图片，必须嵌入，不要假装不存在
- **不搞自娱自乐式推导**——推导是给"什么都不懂的人"看的，每一步都要解释
- **不搞缺斤少两的流程图**——SVG 推导链的每个节点和箭头都要有文字说明
- **不跳过调用链**——技术类内容必须追踪完整的调用链（用户层→系统调用→OS层→硬件），不能只讲一层
- **不省略常见错误**——每个场景至少列出 2-3 个学生最容易犯的错误，用 warning callout
- 不把书外内容混进原文引用
- 不在 §4 里重新构建心智模型——那是 §3 的事，§4 只管填充

### 质量标准

- 每个场景都回扣了 §3 的心智模型（"这对应生长链的哪个节点"）
- 每个场景开头有详细的图景解说（不是 1-2 句话的概括，而是详细展开"这个场景在描述什么"）
- 原文引用覆盖原书所有知识点（自检：把所有 blockquote 拼起来是否覆盖原书每一节）
- 原书有图的地方，HTML 里也有图（路径正确，图片可见）
- 公式推导从零开始，每一步都有文字解释（不只是算式）
- SVG 流程图的每个节点和箭头都有文字说明
- 技术类内容：完整源码 + 完整调用链 + 常见错误 callout
- 覆盖度检查：原书对应章节的每个知识点、定义、公式、例题都在 §4 中出现
- 读者读完 §4 能独立做原书例题（§6 苏格拉底会验证这一点）
- 读者读完 §4 的感受是"我不需要再打开原书了"
- `scripts/validate-html.mjs` 全部通过

## Phase 5 — 费曼讲法

**目标**：用大白话压力测试理解。

**做**：
- 彻底放下书的术语
- 用日常物件、生活场景、对话体
- **个性化原则**：Agent 可选用对话式风格（"你"、"咱们"、"比如你正在……"），但不是强制
  - 判断标准：如果对话式风格能降低理解门槛（比如讲解一个抽象数学概念），就用
  - 如果对话式风格显得轻浮、不符合学科气质（比如严谨的逻辑证明），保持客观风格
  - 依据：Mayer 个性化原则——对话式风格提升学习效果，但不是万能药
- 故意夸张到不严谨
- 结尾必须列"大白话和原书的差别"——哪些地方为了好懂牺牲了精确

**不做**：
- 不做术语翻译（把"哈密顿量"说成"能量算符"不是费曼讲法）
- 不省略"差别"部分——这一段比类比本身更重要

**质量标准**：
- 一个完全不懂这门学科的朋友能听懂至少 60%
- "差别"列表覆盖所有关键牺牲

## Phase 6 — 苏格拉底诘问

**目标**：让用户自己暴露逻辑漏洞。

**做**：

**A. 课前诊断（出题前必做）**：

- 先列出本章 **3–5 个最常见的误解**
- 每个误解包含四要素：
  1. 学生会怎么想错（具体的错误思路）
  2. 为什么这个想法很自然（看起来合理的地方）
  3. 真相是什么（正确的理解）
  4. 对照案例（一个具体例子展示对 vs 错的差别）
- 依据：PEBBLE 框架——诊断性提问和误解修复是所有 LLM 辅导系统的最大短板
- 误解来源：优先从原书"常见错误"段落提取，其次从学科教育研究文献推理，最后从 Agent 知识推断

**B. 出题**：

- 4–8 道使用题（不是背诵题）
- 用 `<details><summary>` 折叠答案
- **每题答案包含完整的错答模式解析**：
  - 正确答案
  - 如果你答错了，最可能的原因是什么（列出 1–2 种典型错误思路）
  - 每种错答模式对应课本哪一节——"如果你选了 B，你可能混淆了 X 和 Y，回去看 §Z"
- 至少有一道"故意挖坑"题——看似合理但违反某个假设
- 难度递进：Q1 戳 §3 核心 → Q2 戳 §4 细节 → Q3+ 边界/反例

**不做**：
- 不出"X 的定义是什么"——背得出不代表懂
- 不超过 8 题——太多用户会跳过
- 不编造不存在的误解——如果原书没有提及常见错误，至少基于学科逻辑推断 3 个

**质量标准**：
- 每题都要求用户**使用**概念，而不是**描述**概念
- 每题答案包含完整的错答模式分析（正确 → 错因 → 定位）
- 误解诊断覆盖本章主要易错点
- 至少一道题覆盖 §3.5 的边界情况（"为什么推导链不能更短"的实战版）

## Phase 7 — 闭环验真

**目标**：用**实际输出**检验理解。不是"合上书写3个核心点"——那是理论。真正的检验是：**你能用学到的东西解决实际问题吗？**

### 核心原则：输出必须是"可验证的行动"

不同学科的输出方式不同，但标准一样：**做出来，不是说出来。**

| 学科类型 | 输出方式 | 说明 |
|---|---|---|
| **基础理论**（数学、物理、概率论、线性代数） | **做考试题** | 出 2-3 道考研/期末考试难度的题，要求学生合上书独立完成。不是 §6 的"使用题"，是**正式考试题**——有标准答案、有评分标准 |
| **技术/工程**（编程、CPU 设计、系统架构） | **写代码或指出源码位置** | 给一个具体任务，要求学生写代码实现，或者在真实代码库中指出对应模块的位置 |
| **概念/人文**（历史、哲学、社科） | **写分析或论证** | 给一个开放性问题，要求学生用本章概念写出有论据的分析 |

### 做

**A. 出实战题（理论学科）**

- 2-3 道题，难度对标**考研真题或期末考试**
- 题目来源优先级：
  1. 原书习题（如果有难度合适的）
  2. 考研/期末真题（agent 从知识库推断）
  3. 基于本章知识点构造的综合题
- 每道题包含：
  - 完整题目
  - **折叠的标准答案**（用 `<details>` 包裹）
  - 评分标准（哪几分来自哪个步骤）
  - 常见错误和扣分点
- **不要提示**：题目旁边不给"这道题考的是 §X.Y"——考试不会告诉你考什么

**B. 出实战题（技术学科）**

- 一个具体任务，要求学生：
  - 写代码实现本章学到的某个概念/算法/机制
  - 或者在真实代码库中指出"这个概念在源码的哪个文件、哪个函数"
- 任务要求：
  - 有明确的验收标准（代码能跑通 / 能指出正确位置）
  - 不是"伪代码"——要求能实际运行的代码
  - 如果是指源码，给出一个真实项目的路径让学生定位

**C. 回到清单核对（保留，但简化）**

- 回到 §1 清单，快速扫一遍：每条标记"✅ 能做题 / ❌ 还不行"
- 不需要逐条复述——做题做出来了就是最好的证明

**D. 下一步建议**

- 哪道题做错了 → 指出回去看哪个 §4 场景
- 全做对 → 进下一章
- 建议用户在 `notes/` 写错题笔记

**E. 跨章导航和进度统计（保留）**

- **跨章导航**：课件底部渲染导航条
  - 格式：`← 上一章：[章节名] | [回目录] | 下一章：[章节名] →`
  - 如果上一章/下一章的课件已存在于 `output/`，链接到对应 HTML
  - 如果不存在，显示灰色占位文字（"下一章课件尚未生成"）
  - 导航条 HTML 结构定义在 `renderers/html-shell.md`
- **学习进度统计**：课件底部渲染进度卡片
  - 格式：已完成 X/Y 章（Z%）| 累计知识点：N 个 | 生成时间：YYYY-MM-DD HH:MM
  - 统计依据：`output/` 目录中已有课件的数量 + §1 知识清单的条目数
  - Agent 在 §7 生成时扫描 `output/` 目录获取统计数据

### 不做

- 不出"用自己的话复述"——说得出不代表做得出
- 不跳过——即使用户觉得"我已经会了"
- 不要在题目旁边提示"这道题考的是什么"——考试不会告诉你
- 不出太简单的题——§6 已经出过使用题了，§7 是**检验**，不是**练习**

### 质量标准

- 理论学科：2-3 道考研/期末难度的题，有标准答案和评分标准
- 技术学科：一个可执行的任务（写代码或指出源码位置），有明确验收标准
- 做题结果直接决定"进下一章"还是"回去补"
- 导航链接有效（如果对应课件存在）
- 进度统计数字准确（与实际 `output/` 文件数一致）
