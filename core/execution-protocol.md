# 执行协议

Agent 执行讲课模式时必须遵守的 5 条协议。规定**怎么做**——分批策略、暂停通告、验证时机、断点恢复、有界生成。

---

## 协议 1：分批输出

**规则**：单次输出不超过 3 个 Phase。写完暂停，等用户确认再继续。

**原理**：CTML 分段原则——分批呈现降低认知负荷。上下文窗口限制——单次生成过多内容导致质量塌方。用户确认机制——让用户有机会在中途纠正方向。

**推荐分批**：

| 内容量 | 分批策略 |
|--------|---------|
| 标准内容（知识点 ≤ 20） | §1–§3 → §4 → §5–§7 |
| 内容量较大（知识点 > 20） | §1–§2 → §3–§4 → §5–§7 |
| 极简内容（知识点 < 8，用户选"入门"深度） | §1–§4 → §5–§7 |

**每批特征**：

- **§1–§3（或 §1–§2）**：建立完整心智框架——知道有什么、能干什么、为什么需要
- **§4（或 §3–§4）**：核心精读——最吃上下文窗口的阶段，通常独立一批
- **§5–§7**：理解验证——费曼 + 苏格拉底 + 闭环，检验前面建立的框架

---

## 协议 2：暂停通告

**规则**：每次暂停时明确告诉用户当前状态、下一步、该检查什么。

**格式**：

```
---
✅ §1–§3 已完成。
📋 下一批：§4 逐字精读（预计 X 个知识点，Y 个公式）。
⚠️ 确认前请检查 §3 的推导链——如果有方向性错误，现在纠正成本最低。
回复"继续"推进到 §4。
---
```

**反例（不要这样）**：

```
上半部分写完了，继续吗？
```

——没有告诉用户"写完了什么"、"接下来是什么"、"现在该检查什么"。

---

## 协议 3：验证时机

**规则**：以下节点必须运行确定性验证，不能只靠 Agent 目测。

| 时机 | 运行什么 | 依赖 |
|------|---------|------|
| §4 完成后 | `scripts/validate-html.mjs` | 检查 W3C 合规性 + 可访问性 + 图片引用 |
| 全部 Phase 完成后 | 三层自检（见 `quality-standards.md`） | 局部 → 上下文 → 全局 |
| 最终输出前 | 手动检查 | HTML 在浏览器打开，检查渲染效果 |

**注意**：

- `validate-html.mjs` 检测到的错误必须修正后才能进入 §5
- 三层自检中任何 ❌ 必须修正后重新自检
- 最终浏览器检查发现渲染问题，回退到对应 Phase 修正

---

## 协议 4：断点恢复

**规则**：磁盘上 `output/.<topic-slug>.progress` 是唯一断点记录。每完成一个 Phase 后追加一行 `phase=<N> timestamp=<ISO-8601>`。全部 Phase 完成后删除 `.progress`。

**恢复逻辑（新会话启动时）**：

1. 检查 `output/.<topic-slug>.progress` 是否存在
2. 如果存在且对应 HTML 已部分生成：
   - 读取 `.progress`，确认最后完成的 Phase
   - 提示用户"上次已完成 §X，从 §Y 继续？"
   - 用户确认 → 跳过已完成 Phase，从断点继续
   - 用户拒绝 → 删除 `.progress`，重新开始
3. 如果 `.progress` 不存在 → 全新开始

---

## 协议 5：有界生成（§4 适用）

**规则**：§4 逐字精读是 HTML 错误率最高的阶段（大量的图文交织 + 公式 + callout）。采用"规划 → 验证 → 渲染"分离模式。

**流程**：

```
1. Agent 分析 §4 内容 → 输出结构化 JSON（规划层）
   - 每个 section 的类型（definition / derivation / example / callout）
   - 每个 section 的数据（文本、公式、图片路径、callout 类型）

2. Agent 检查 JSON schema 完整性（验证层）
   - 所有必填字段就绪
   - 图片路径可解析
   - callout 类型合法

3. 运行 scripts/render-courseware.mjs 读取 JSON → 生成 §4 的 HTML 片段（渲染层）

4. Agent 将生成的 HTML 片段嵌入主模板
```

**好处**：

- HTML 片段由经过测试的模板生成，不会出现标签不闭合、CSS 冲突
- Agent 只需关注内容正确性，不需同时关注 HTML 语法
- JSON 比 HTML 更容易被校验和调试

**适用范围**：

- 初始仅应用于 §4（内容密度最高、HTML 结构最复杂）
- 稳定后可扩展至 §1、§5、§6
- §3 和 §7 保留自由格式（SVG 图 + 较简文本，JSON 化的收益低于成本）
