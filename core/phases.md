# 七阶段定义

骨架不许调换顺序。每一阶段的 HTML 结构和样式约定见 `renderers/html-shell.md`，完整模板见 `templates/`。

**设计依据见 `core/learning-science.md`**：CTML 预训练 → §1；引导发现 + 信号 → §3；脚手架 → §4；个性化 + 生成活动 → §5–§7。

---

## Phase 1 — 知识点清单 + 知识地图

**目标**：让用户在读具体内容之前就知道"这章有多少东西、它们之间什么关系"。

**做**：
- 穷尽本章所有知识点（定义/公式/定理/例题/图/脚注）
- PPT 出现过的用 🎯 标记
- **每个知识点附一句"一句话定义"** —— 不展开，只先命名。格式：`1. [知识点名]（原书 §X.Y）🎯 — 一句话定义。`
  - 依据：CTML 预训练原则——先命名概念再深入讲解，学习者后续加工效率更高
  - 一句话定义标准：用日常语言，不包含多于 1 个术语；如果必须用术语，紧接着用括号解释
- 清单之后画一张 SVG 知识地图，展示所有知识点的层级和依赖关系
- 地图中每个节点需标注：上游（依赖什么）、下游（打开什么）、邻居（并列什么）

**不做**：
- 不跳过任何原书脚注或边栏注释
- 不省略"简单"的概念——VSL 用户需要全局视野
- 一句话定义不展开——展开是 §3–§5 的事

**质量标准**：
- 清单条目数 ≥ 原书小节数（通常更多，因为一个小节可能有多个概念）
- 知识地图覆盖清单所有条目
- 每个知识点都有一句话定义
- 一句话定义不包含多于 1 个术语

## Phase 2 — 逆向目标

**目标**：让用户知道"学完能干什么"，而不是"要学什么"。

**做**：
- 从例题/习题/PPT 练习中提取 3–6 条可检验的能力陈述
- 每条格式："给定 [输入]，能 [动作]，并且 [质量约束]"
- 必须对应到原书具体出处

**不做**：
- 禁止"理解/掌握/了解/熟悉"这类不可检验动词
- 不超过 6 条——太多记不住

**质量标准**：
- 覆盖本章主要例题类型
- 每条能力陈述都可以用一道题验证

---

## Phase 3 — 第一性原理（Why：为什么需要这个概念）

**目标**：回答"这个概念为什么必须存在"——不是"这个概念是什么"。§3 只管动机和起源，具体内容留给 §4。

**做**：
- §3.1 最初的问题（优先从读者体验切入，其次例题，最后学科史）——给出**动机**
- §3.2 最小核心（一句话：本章只在做一件什么事）
- §3.3 一砖一瓦垒起来（假设 → 推出 → 加假设 → 推出 → **停在概念诞生的那一刻**）
  - **每个推导步骤标注序号**（Step 1/n），关键转折步骤紫色高亮标注"🔑 关键转折"
  - 依据：Mayer 信号原则——步骤编号降低视觉搜索成本，迁移测试得分提升 d=0.58
- 附 SVG 推导链图，图中节点与 §3.3 正文步骤序号一一对应
- §3.4 在知识框架中的位置（SVG 树形图 / 对照表）
- §3.5 为什么这条链不能更短（至少去掉一个假设，展示会发生什么）
- **结尾插入"暂停思考"框**（元认知提示）：

```
> 🧠 **暂停思考**
> 
> 现在合上书（或者滚到屏幕外面去），问自己：
> - 如果让你跟一个没学过这章的朋友讲"为什么需要 X"，你能讲清楚吗？
> - §3.3 的推导链里，哪一步你觉得"太跳跃了"？
> 
> 如果都卡住了，把对应的那一步重新读一遍再继续。
> —— §4 将展开具体怎么用、怎么做题。但先确保"为什么"是稳的。
```

**不做**：
- 不宣布结论——要推出结论
- 不跳步——跳了就断了画面
- 不展开公式怎么用、怎么做题——这些是 §4 的事
- 不重复 §4 会讲的细节推导——§3.3 止于"概念诞生"，§4 才展开"概念落地"

**质量标准**：
- §3.3 最后一句话读起来像"……所以这个概念不可避免地出现了"
- 推导链图中每个节点都能在 §3.3 正文中找到对应推导（且序号匹配）
- 读者读完 §3 的感受是"哦，原来不得不发明这个东西"，而不是"哦，这个东西怎么算"

## Phase 4 — 逐字逐句精读（What+How：这个概念是什么、怎么用）

**目标**：落到原书每一个定义、公式、例题上。§3 给了动机，§4 建立做题能力。

**做**：
- "引一段—讲一段"循环——每段原文引用后必须跟讲解，用自己的话把原文"翻译"成可操作的理解
- 原书有图就用 `<img>` 嵌入，紧跟在引出它的那句话后面（图文交织）
- 书外补充用蓝色 callout，PPT 补充用橙色 callout
- 多步骤例题配 SVG 过程图
- **多步骤例题/公式推导中，每个关键步骤标注序号**（Step 1/n）
  - 依据：Mayer 信号原则，同 §3
- 公式必须推导或解释每一步的意义，不能只列结果
- 每个定义至少配一个具体例子和一个反例
- **有界生成流程**：§4 内容密度最高、HTML 结构最复杂，采用分离模式
  - Agent 先输出结构化 JSON → 检查 schema → `scripts/render-courseware.mjs` 渲染 HTML 片段 → 嵌入主模板
  - JSON schema 定义见 `scripts/render-courseware.mjs` 文件头部注释
- **§4 完成后强制运行** `scripts/validate-html.mjs`，所有错误修正后进入 §5
- **结尾插入"暂停思考"框**（元认知提示）：

```
> 🧠 **暂停思考**
> 
> 你刚读完本章最硬核的部分。在进入费曼讲法（用大白话重新讲一遍）之前：
> - 不看屏幕，用自己的话说出本章核心公式/定理——说一遍就行，不用精确。
> - 如果卡住了，是你刚刚跳读了，还是某个推导步骤真的没跟上？
> - 卡在推导步骤 → 回去重新看那一步。
> - 卡在"记不住" → 正好，§5 费曼讲法就是来解决这个的。
```

**不做**：
- 不大段复述原书——原书已经说了的不用换种说法重复
- 不把书外内容混进原文引用
- 不跳过任何原书的推导过程——跳了读者做题时会断

**质量标准**：
- 每段原文引用后都有对应的讲解
- 每个 callout 都标注了来源（书外/PPT）
- 读者读完 §4 能独立做原书例题（§6 苏格拉底会验证这一点）
- `scripts/validate-html.mjs` 全部通过

## Phase 5 — 费曼讲法

**目标**：用大白话压力测试理解。

**做**：
- 彻底放下书的术语
- 用日常物件、生活场景、对话体
- **个性化原则**：Agent 可选用对话式风格（"你"、"咱们"、"比如你正在……"），但不是强制
  - 判断标准：如果对话式风格能降低理解门槛（比如讲解一个抽象数学概念），就用
  - 如果对话式风格显得轻浮、不符合学科气质（比如严谨的逻辑证明），保持客观风格
  - 依据：Mayer 个性化原则——对话式风格提升学习效果，但不是万能药
- 故意夸张到不严谨
- 结尾必须列"大白话和原书的差别"——哪些地方为了好懂牺牲了精确

**不做**：
- 不做术语翻译（把"哈密顿量"说成"能量算符"不是费曼讲法）
- 不省略"差别"部分——这一段比类比本身更重要

**质量标准**：
- 一个完全不懂这门学科的朋友能听懂至少 60%
- "差别"列表覆盖所有关键牺牲

## Phase 6 — 苏格拉底诘问

**目标**：让用户自己暴露逻辑漏洞。

**做**：

**A. 课前诊断（出题前必做）**：

- 先列出本章 **3–5 个最常见的误解**
- 每个误解包含四要素：
  1. 学生会怎么想错（具体的错误思路）
  2. 为什么这个想法很自然（看起来合理的地方）
  3. 真相是什么（正确的理解）
  4. 对照案例（一个具体例子展示对 vs 错的差别）
- 依据：PEBBLE 框架——诊断性提问和误解修复是所有 LLM 辅导系统的最大短板
- 误解来源：优先从原书"常见错误"段落提取，其次从学科教育研究文献推理，最后从 Agent 知识推断

**B. 出题**：

- 4–8 道使用题（不是背诵题）
- 用 `<details><summary>` 折叠答案
- **每题答案包含完整的错答模式解析**：
  - 正确答案
  - 如果你答错了，最可能的原因是什么（列出 1–2 种典型错误思路）
  - 每种错答模式对应课本哪一节——"如果你选了 B，你可能混淆了 X 和 Y，回去看 §Z"
- 至少有一道"故意挖坑"题——看似合理但违反某个假设
- 难度递进：Q1 戳 §3 核心 → Q2 戳 §4 细节 → Q3+ 边界/反例

**不做**：
- 不出"X 的定义是什么"——背得出不代表懂
- 不超过 8 题——太多用户会跳过
- 不编造不存在的误解——如果原书没有提及常见错误，至少基于学科逻辑推断 3 个

**质量标准**：
- 每题都要求用户**使用**概念，而不是**描述**概念
- 每题答案包含完整的错答模式分析（正确 → 错因 → 定位）
- 误解诊断覆盖本章主要易错点
- 至少一道题覆盖 §3.5 的边界情况（"为什么推导链不能更短"的实战版）

## Phase 7 — 闭环验真

**目标**：用主动输出检验整条链上有没有断点。不是"回顾清单"，是"合上书复述"。

**做**：
- **主动输出任务**：要求用户合上书，用自己的话写出本章 3 个核心点（不是背定义，是用自己的语言讲清楚）
- 回到 §1 清单逐条核对：每条问"我能合上书把它讲一遍吗？"
- 给出下一步建议（哪些卡了就回去看哪、整体通了就进下一章）
- 建议用户在 `notes/` 写理解笔记
- **跨章导航**：课件底部渲染导航条
  - 格式：`← 上一章：[章节名] | [回目录] | 下一章：[章节名] →`
  - 如果上一章/下一章的课件已存在于 `output/`，链接到对应 HTML
  - 如果不存在，显示灰色占位文字（"下一章课件尚未生成"）
  - 导航条 HTML 结构定义在 `renderers/html-shell.md`
- **学习进度统计**：课件底部渲染进度卡片
  - 格式：已完成 X/Y 章（Z%）| 累计知识点：N 个 | 生成时间：YYYY-MM-DD HH:MM
  - 统计依据：`output/` 目录中已有课件的数量 + §1 知识清单的条目数
  - Agent 在 §7 生成时扫描 `output/` 目录获取统计数据

**不做**：
- 不跳过——即使用户觉得"我已经会了"
- 不要只列清单不检验——清单是工具，输出才是目的

**质量标准**：
- 清单每一条都有对应的检验方式
- 包含至少一个主动输出任务（不是被动阅读）
- 导航链接有效（如果对应课件存在）
- 进度统计数字准确（与实际 `output/` 文件数一致）
